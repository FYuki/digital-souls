# テスト方針

## 基本方針

`digital-souls` のテストは、単体テストと結合テストを明確に分ける。

- **単体テスト**: モックを用いて、個々の関数・クラス・コンポーネントのロジックを検証する
- **結合テスト**: 実際の外部サービスに接続して検証する（モック禁止）

単体テスト（モック）のみでは、機能の完了条件の実証として扱わない。結合テストによる実接続の検証をもって、初めて「実際に動作する」ことの一次証跡とする。

## 対象となる外部サービス

現時点で結合テストの対象となる外部サービス:

- Ollama（LLM推論）
- ChromaDB（RAGベクトルストア）
- VOICEVOX（TTS）
- Whisper（STT）

BE/FEの結合を検証する場合は、BE側も実プロセスとして起動した上で接続する（BEをモックしたFEテストは結合テストと呼ばない）。

## 命名・配置

結合テストは、ファイル名・配置から「結合テストである」と分かるようにする。BE/FEで採用しているテストランナーが異なるため、命名規則はそれぞれの慣習に従う。

- **BE（pytest）**: `backend/tests/test_<対象>_integration.py`
  - 単体テスト: `test_<対象>.py`
- **FE（Playwright）**: `frontend/e2e/<対象>.spec.ts`
  - 単体テスト: `frontend/src/**/*.test.ts`
  - `e2e/` 配下は「結合テスト」の置き場として扱う。モックWebSocketバックエンド（`installMockWebSocketBackend` 等）を使っている場合、それは結合テストの要件を満たさないため、次のいずれかにする
    - 実BEプロセスに接続する結合テストとして書き直す
    - モックのままでよい場合は、単体テスト相当として `e2e/` から移すか、ファイル名・テスト名で「モックバックエンドを使用」と明示する

## 結合テストの要件

- 外部サービスへの通信を `unittest.mock.patch`（BE）や `page.route` / モックWebSocket（FE）等でモックしてはならない
- 接続先は環境変数で切り替え可能にする（例: `TEST_OLLAMA_BASE_URL`）
  - 単体テスト・本番実装で使う `OLLAMA_BASE_URL` 等の環境変数とは分離する
  - テスト専用の環境変数が未設定の場合は、そのテストを `skip` する

## ローカル/CIでの実行方針

結合テストは、実接続先をローカルでのみ用意する想定とする。

- **ローカル環境**: 開発者が起動済みの実サービス（Ollama等）にテスト専用の環境変数（`TEST_OLLAMA_BASE_URL` 等）で接続し、結合テストを実行する
- **CI環境**: テスト専用の環境変数が未設定であることを前提とし、結合テストは自動的にスキップする（BE: `pytest.mark.skipif` 等、FE: `test.skip` 等）
  - CI上での実サービスコンテナ起動（Ollama等）は現時点では行わない
  - そのため、結合テストの実行・確認は開発者がローカルで責任を持って行う

## 完了条件レポートにおける扱い

PR等で「動作確認済み」と報告する場合、結合テストの実行ログ（実接続によるレスポンス内容を含む）を一次証跡として添付する。単体テストの通過のみを根拠にしない。
