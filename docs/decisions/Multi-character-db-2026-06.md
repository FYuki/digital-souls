# 複数キャラクター運用時のDB構成・ハードウェア限界 議事録 (2026-06)

## 概要

AITuber/AIキャラクター向けRAG構成の標準パターンを起点に、`digital-souls` で複数キャラクターを並行運用する場合のDB構成（Chroma/SQLite vs Qdrant/PostgreSQL）と、Mac mini M4 Pro 48GBでのハードウェア限界について検討した。

決定事項・設計指針は `docs/decisions/multi-character-db-2026-06.md` に反映済み。本ファイルは検討の過程をやり取り単位で残す議事録。

---

## 論点1: AITuber RAGの標準構成

**Q. AIVTuberのRAGの構成としてスタンダードはどんな構成？**

AITuber向けRAGは汎用チャットボットRAGと異なり、配信のリアルタイム性・キャラクター性の一貫性・長期的な関係性の記憶が重要になる。標準構成は以下の3層。

- 埋め込み・ベクトル検索層（Ollama経由のnomic-embed-text等）
- メモリの階層構造（短期記憶／エピソード記憶／長期・性格記憶）
- 検索戦略（時間減衰スコアリング、視聴者IDフィルタリング、Character Card固定コンテキストとの併用）

ベクトルDBは開発フェーズで変える想定。MVPはChroma/SQLite、本番・スケール要件が出たらQdrant/PostgreSQLへ移行する構成が一般的。

**→ digital-soulsのPhase 1（Chroma/SQLite）→Phase 3（Qdrant/PostgreSQL）という既存方針は、この標準パターンに沿っている。**

---

## 論点2: Chroma/SQLite と Qdrant/PostgreSQL の違い

**Q. Chroma/SQLite、Qdrant/PostgreSQLの違いは？**

| 観点 | Chroma/SQLite | Qdrant/PostgreSQL |
|---|---|---|
| 性質 | Python内包型ベクトルDB＋組み込みファイルDB | 専用ベクトル検索サーバー＋汎用RDBMS |
| 運用の手間 | プロセス管理ほぼ不要、WSL2無Docker方針に合う | 別プロセスの起動・監視・バックアップが必要 |
| スケーラビリティ | 数十万〜数百万件規模までは問題なし | 高QPS・大規模データ・分散構成に強い |
| 検索機能 | フィルタリング可能だが大規模では劣る | ペイロードフィルタリング最適化、複合条件検索が高速 |
| 永続化・耐障害性 | ファイルベース | トランザクション保証（MVCC）あり |

## 論点3: どれぐらいの運用まで耐えられるか（単一キャラクター前提）

**Q. どれぐらいの運用まで耐えられる？**

- **Chroma**: 数万〜数十万件のベクトルまでは快適。100万件超でレイテンシが秒単位に伸び始める。個人運営の配信規模（週5日×1年で十数万件程度）なら、要約・圧縮処理を入れれば長期運用でも対応可能な範囲
- **SQLite**: 「ライターは同時に1つまで」の制約。低頻度・逐次書き込みなら問題ないが、高頻度の同時書き込みには向かない。ファイルサイズは実用上数GB〜数十GB程度に収まることが多い
- **移行を考えるべきサイン**: 検索レスポンスの遅延を体感する、複数キャラクター・チャンネルを並行運用したくなる、DBロックエラーや破損が発生する、視聴者数の大幅増加

**→ 単一キャラクター・個人運用なら、Chroma/SQLiteは数年単位の運用でも十分に耐えうるという結論。**

---

## 論点4: 複数キャラクター・並行書き込み前提への変更

**Q. 複数キャラクターでの会話想定があるため、並行書き込みを前提としています。**

前提が変わったことで、SQLiteのライター制約とChromaのプロセス内動作という制約が早期に効いてくる。検討した方向性は2つ。

- **方向性A**: キャラクターごとにDBを分離し、プロセスレベルで完全独立させる。キャラクター間でメモリ共有が不要ならChroma/SQLiteのMVP構成を維持できる
- **方向性B**: キャラクター間で共有メモリが必要なら、最初からPostgreSQL（pgvector）/Qdrantに進む

判断基準として「キャラクター間でメモリ・会話コンテキストを共有する設計か」を確認。

**A. キャラごとに独立のほうが良い。**

**→ 方向性Aを採用。各キャラクターは完全に独立したメモリ空間を持つ。**

---

## 論点5: 複数インスタンス並行稼働の弊害とMac mini M4 Pro 48GBでの実現性

**Q. 複数インスタンスを立てることの弊害は？また、本番環境はMacmini M4pro 48GB想定です。**

弊害として以下を整理。

1. **メモリ（RAM）の重複消費**: 埋め込みモデルをプロセスごとに個別ロードすると、キャラクター数分メモリを消費する。Ollamaを共有サーバー化しAPI経由で呼び出す構成で回避
2. **ファイルロックの競合リスク**: ディレクトリ構成を `characters/{id}/db/` 等に明確分離する必要
3. **プロセス管理の複雑化**: キャラクター数増加でsystemd等のプロセス管理ツールが実質必須になる
4. **ディスクI/Oの競合**: 配信中のRAG検索とTTSが同時に走るとI/O待ちが発生する可能性

Mac mini M4 Pro 48GBでの見立てとして、Ollama共有サーバー化により2〜3キャラクター程度の同時稼働は現実的な範囲に収まるが、キャラクターごとに別LLMインスタンスを起動する構成だとモデルサイズ×キャラクター数で48GBに近づくリスクを指摘。GPU（Metal）の計算競合（推論のキューイング）もこの時点で言及。

## 論点6: Chroma/SQLite構成とQdrant/PostgreSQLの負荷分岐点

**Q. この構成と、Qdrant/PostgreSQLの負荷分岐点はある？**

3つの軸で分岐点を検討。

- **軸1（キャラクター数×同時アクセス）**: 4〜5体を超え、同時発話が頻発する場合、DBファイル管理の運用コストがPostgreSQL集約の移行コストを上回るタイミングが分岐点。技術的限界より先に運用上の分岐点が来る
- **軸2（メモリ件数とレイテンシ）**: キャラクター単位で独立しているため、各キャラが数万件程度ならデータ量起因の分岐点には到達しない。ただしキャラクター間共通知識ベースが必要になった場合は別途検討が必要
- **軸3（書き込み頻度とMac mini固定リソースの競合）**: SQLite/Chromaの限界より先に、Mac mini上のCPU/GPU/メモリの物理的競合が表面化する可能性が高いと指摘

**結論（この時点）**: LLM推論・音声合成のリソース競合が最初に来る可能性が高く、運用コストの増大（5体以上が目安）がその次、データ量起因のレイテンシ悪化は個人運営では到達しにくいという優先順位を提示。

## 論点7: LLMサイズとTTS構成を確定した上での再評価

**Q. キャラクターLLMはe4b多くても12b、複数会話時はVOICEVOX複数インスタンスまたは、他SST組み合わせの場合でもハードウェアの限界が先ですか？**

前提を確定（e4b〜12b、VOICEVOX複数または他TTS併用）した上でメモリ予算を再計算。

| 項目 | 見積もり |
|---|---|
| LLM（e4b + 12b同時ロード） | 12〜13GB程度 |
| VOICEVOX複数インスタンス（3〜4） | 3〜4GB程度 |
| 各キャラクターのChroma/SQLite | 数GB程度 |
| OS・常駐プロセス | 数GB |
| **合計目安** | **25〜30GB程度（48GBに対し余裕あり）** |

**→ この前提では、メモリ容量の限界はほぼ問題にならない。**

先に効いてくるのは以下の2点と結論。

1. GPU推論のキューイングによる応答タイミングのズレ（複数キャラ同時発話時、数百ms〜1秒程度）
2. OS・プロセス管理の複雑さ（運用コスト、技術限界ではなく設計・運用上の課題）

DB層（Chroma/SQLite vs Qdrant/PostgreSQL）の分岐点は、このモデルサイズ・構成では当面到達しない可能性が高いと結論。対策としてはGPU推論のキューイング順序制御、軽量モデルへの切り替え、一部キャラクターのクラウドAPI分離（Claude Haiku等）を挙げた。

## 論点8: クラウド移行時の単一DB vs 複数インスタンス

**Q. 運用が煩雑化した際に、クラウドサービスを利用することになった場合、1つのDBか、複数インスタンスに分かれているかは課題になる？**

| 観点 | 単一DB（統合） | 複数インスタンス（キャラクターごと） |
|---|---|---|
| 移行のしやすさ | `character_id`があれば比較的スムーズ | ローカル構成と1対1でそのまま移行可能 |
| コスト効率 | キャラクター数が増えてもコストは線形に増えにくい | インスタンス単位の最低料金がキャラクター数分発生しやすい |
| 運用の一元化 | バックアップ・監視・スキーマ変更が1箇所で済む | キャラクターごとに個別管理が必要 |
| 障害耐性 | 単一障害点になりうる | あるキャラクターの障害が他に波及しない |

**結論**: 技術的な移行作業自体はどちらでも大きな障害にはならないが、コストと障害耐性のトレードオフが実運用上の課題になる。個人運営・収益化前なら単一DB＋`character_id`の論理分離が現実的。収益化後、重要キャラクターのみ独立インスタンス化する段階的対応も選択肢。

**今後の備えとして**: ローカル開発段階から全レコードに `character_id` フィールドを必ず持たせておくことを推奨。将来どちらの構成に進む場合でもスムーズに対応でき、後から追加するマイグレーションコストを避けられる。

---

## 決定事項サマリー

1. キャラクター間でメモリ・会話コンテキストは共有せず、完全に独立させる
2. Chroma/SQLiteのMVP構成を維持する（Qdrant/PostgreSQLへの早期移行は不要と判断）
3. 全レコードに `character_id` を付与する設計をPhase 3のRAGスキーマ設計時点で組み込む
4. 埋め込みモデル・LLM推論・VOICEVOXは共有サーバー化し、各キャラクタープロセスはAPI経由で呼び出す
5. ハードウェア（Mac mini M4 Pro 48GB）の制約は、DB層よりGPU推論のキューイングが先に表面化する見立てのため、優先課題はGPU推論の順序制御・一部キャラクターのクラウドAPI分離の検討に置く

## 保留事項（Phase 3で確認）

- 複数キャラクター同時発話時のGPU推論キューイング設計
- `character_id` を含めたRAGスキーマの具体的なテーブル/コレクション設計
- Mac mini M4 Pro 48GB上での実機検証（メモリ・GPU負荷の実測）
